# Interesting Reads in Software Engineering

## Article Link 

[**Write code that is easy to delete, not easy to extend**](https://programmingisterrible.com/post/139222674273/write-code-that-is-easy-to-delete-not-easy-to)

## What I found interesting 

This article really clicked with me because it challenges a supper common instinct in software engineering; designing syatems to be endelessly extensible from the start.The author argues that a lot of future proofing can actually make code harder to work with as it adds abstractions and complexity before you truly know what the future needs are. Instead the article suggests optimizing for deletion writing code in a way that makes it easy to remove , replace or rewrite when requirements change(which they almost always do).

## Takeaway

If I had to summarize the lesson: **simplicity + low coupling beats speculative abstraction**. Codebases stay healthier when theyâ€™re built to evolve through real feedback rather than predicted futures.
